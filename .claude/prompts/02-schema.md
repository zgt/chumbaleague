# Step 2: Database Schema

Add the full music league schema to `packages/db/src/schema.ts`. This is a fresh database — we only need music league tables plus the auth tables (which already exist in `auth-schema.ts`).

## What to do

### 1. Update `packages/db/src/auth-schema.ts`

The `user` table needs a `notificationPreferences` jsonb column. Update the user table to include:

```ts
import { boolean, index, jsonb, pgTable, text, timestamp } from "drizzle-orm/pg-core";

export const user = pgTable("user", {
  id: text("id").primaryKey(),
  name: text("name").notNull(),
  email: text("email").notNull().unique(),
  emailVerified: boolean("email_verified").default(false).notNull(),
  image: text("image"),
  notificationPreferences: jsonb("notification_preferences")
    .$type<{
      roundStart: boolean;
      submissionReminder: boolean;
      votingOpen: boolean;
      resultsAvailable: boolean;
    }>()
    .default({
      roundStart: true,
      submissionReminder: true,
      votingOpen: true,
      resultsAvailable: true,
    })
    .notNull(),
  createdAt: timestamp("created_at").defaultNow().notNull(),
  updatedAt: timestamp("updated_at")
    .defaultNow()
    .$onUpdate(() => new Date())
    .notNull(),
});
```

Keep `session`, `account`, and `verification` tables as-is.

**IMPORTANT**: Since the auth schema is auto-generated by `@better-auth/cli`, you may need to also update the auth config to include the custom `notificationPreferences` field, or manually add it after generation. The field is NOT managed by better-auth — it's a custom addition. Make sure the `generate` script output doesn't overwrite it. Consider adding a comment noting this.

### 2. Replace `packages/db/src/schema.ts`

Remove the scaffold `Post` table and add all music league tables. Reference `~/coding/todo-list/packages/db/src/schema.ts` for the exact table definitions.

Tables to include (copy from tokilist):

**Enums:**
- `leagueStatusEnum` — ACTIVE, COMPLETED, ARCHIVED
- `memberRoleEnum` — OWNER, ADMIN, MEMBER
- `roundStatusEnum` — PENDING, SUBMISSION, LISTENING, VOTING, RESULTS, COMPLETED
- `reportStatusEnum` — PENDING, REVIEWED, DISMISSED
- `reportReasonEnum` — SPAM, OFFENSIVE, HARASSMENT, OTHER
- `contentTypeEnum` — LEAGUE, SUBMISSION, USER, COMMENT, ROUND (remove TASK — that's tokilist-specific)

**Core Music League Tables:**
- `League` — name, description, settings (songsPerRound, maxMembers, allowDownvotes, voting config, etc.), inviteCode, creatorId
- `LeagueMember` — leagueId, userId, role, joinedAt
- `Round` — roundNumber, themeName, themeDescription, status, dates (start, submission deadline, voting deadline), playlistUrl, leagueId
- `Submission` — spotifyTrackId, track metadata (name, artist, album, art URL, preview URL, duration), roundId, userId
- `Vote` — points, roundId, voterId, submissionId
- `Comment` — text, submissionId, userId

**Moderation Tables:**
- `Report` — reporterId, reportedUserId, contentType, contentId, reason, details, status
- `BlockedUser` — userId, blockedUserId
- `ContentFlag` — contentType, contentId, flaggedText, matchedWords

**Supporting Tables:**
- `PushToken` — userId, token, platform
- `ThemeTemplate` — name, description, category

**Relations (all of them):**
- `leagueRelations` — creator (user), members (LeagueMember[]), rounds (Round[])
- `leagueMemberRelations` — league (League), user (user)
- `roundRelations` — league (League), submissions (Submission[])
- `submissionRelations` — round (Round), user (user), votes (Vote[]), comments (Comment[])
- `voteRelations` — round (Round), voter (user), submission (Submission)
- `commentRelations` — submission (Submission), user (user)
- `userRelations` — leagues (League[]), leagueMemberships (LeagueMember[]), submissions (Submission[]), votes (Vote[]), comments (Comment[])
- `reportRelations` — reporter (user), reportedUser (user)
- `blockedUserRelations` — user (user), blockedUser (user)
- `pushTokenRelations` — user (user)

**Zod Schemas (validation):**
- Do NOT include any task/category/list schemas from tokilist
- Only include music-league-relevant schemas if needed later (can defer to Step 3)
